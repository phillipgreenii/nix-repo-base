**Goal:** Wire the generic `jira` CLI (from SP1 in `phillipg-nix-repo-base`) into
`phillipg-nix-ziprecruiter` so that `jira` is on PATH with ZR credentials resolved automatically;
preserve the `pg-pr-issues-jira-zr` binary name as a back-compat alias; and document the one-time
keychain setup step.

**Architecture:** The ZR repo consumes `repo-base`'s `homeModules.jira` and its overlay (which puts
`pkgs.jira` in scope). A nix-managed TOML config file is written to
`$XDG_CONFIG_HOME/jira/config.toml` with `base_url`, `email`, and
`secret.source = "command"` pointing at `security find-generic-password -s zr-jira -a <email> -w`.
A `writeShellApplication` wrapper named `pg-pr-issues-jira-zr` (installed on PATH in place of the
current `pg-pr-zr`-built binary plus its env-exporting wrapper) forwards all arguments to the real
`jira` binary so `pr-pool`'s shell-out contract is preserved without change.

**Tech Stack:** Nix/home-manager; `pkgs.writeShellApplication`; `pkgs.writeText`; existing
`darwin/system/default.nix` `sharedModules` + `nixpkgs.overlays` wiring.

**Spec:** `docs/superpowers/specs/2026-06-26-generic-jira-access-tool-design.md` §10 item 1
(Bead `pg2-2x2d.2`).

**Precondition:** SP1 MUST be merged to `phillipg-nix-repo-base`'s default branch and the
`phillipgreenii-nix-base` input in this flake's `flake.lock` MUST be bumped to a commit that

## Global Constraints

> Copied verbatim from the spec's binding facts (§1.1 and §8):

- **No ZR string in the GENERIC tool; ZR values live ONLY in this edge repo.** The `jira` binary
  in `phillipg-nix-repo-base` MUST contain no `ziprecruiter`, `zr-jira`, `security`, or other
  tenant/OS string. All such values are supplied here as configuration. This constraint MUST be
  verified by the SP1 guardrail test (`TestNoForbiddenStrings`) before SP2 begins.
- **The config file is generated by nix, not written by hand.** The generated TOML lands at
  `$XDG_CONFIG_HOME/jira/config.toml` (i.e., `~/.config/jira/config.toml`). Because
  `home-manager` manages it via `xdg.configFile`, the file is read-only in the store; the `jira`
  CLI opens it read-only at runtime.
- **The API token MUST NOT enter the nix store.** It is resolved at runtime by the
  `secret.source = "command"` argv (`security find-generic-password -s zr-jira -a <email> -w`).
  The keychain service name `zr-jira` is the unified replacement for the two divergent names used
  by the old tools (`work-activity-tracker-jira` and `activity-collector-jira`). The migration
  of existing entries is a one-time user action (see Task 4).
- **`pg-pr-issues-jira-zr` name MUST be preserved on PATH.** `pr-pool` shells out to
  `pg-pr-issues-jira-zr search --jql … --limit N` and parses the `{items, truncated}` JSON
  envelope. The alias wrapper MUST pass all arguments through unchanged and MUST NOT alter the
  stdout envelope.
- **Permissions ordering:** per `CLAUDE.md`, the `permissions.allow` array in
  `modules/claude-code/settings.local.json` MUST remain in alphabetical order. If this change
  touches that file, the `sort-claude-permissions` pre-commit hook will enforce it automatically.
  Do not sort it manually outside the hook.
- **No `sudo`, no activation commands.** Run `zn-self-build` (or `nix flake check &&
darwin-rebuild check --flake .`) to validate. Ask the user to run `zn-self-apply` manually.
- **LaunchAgents/Daemons:** any new background service MUST go through
  `phillipgreenii.system.launchdServices.{daemons,userAgents}` from `phillipgreenii-nix-personal`
  (ADR 0049). SP2 does not add any new service, so this rule is noted but not exercised here.
- **Markdown formatting:** technical content (globs, paths, identifiers with underscores, cron
  expressions) MUST be wrapped in backticks when authored in `*.md` files (prettier enforces this
  via the `treefmt` pre-commit hook).
- **Validation before "done":** `prek run --all-files` MUST pass; `nix flake check &&
darwin-rebuild check --flake .` MUST pass. Run `zn-self-build` as a convenience alias for the
  latter. NEVER run `zn-self-apply` or `zn-self-upgrade` in a sandboxed environment.

darwin module (confirmed at `darwin/system/default.nix:28-31`), and its value flows into
home-manager's `extraSpecialArgs` indirectly via the system config. The safest approach for a
home-manager module is to receive the email as a literal (or read
`config.home.username`-derived identity if exposed); look up the actual option path in the
personal module. **If the identity option is not reachable from within a home-manager module,**
declare the email as a module option with a default of `"phillipg@ziprecruiter.com"` and set it
at the call site in `home/ziprecruiter/jira/default.nix`.

The TOML template (what `~/.config/jira/config.toml` will contain):

```toml

email    = "phillipg@ziprecruiter.com"

default_limit = 100
```

- [ ] **Step 1: Create `home/ziprecruiter/jira/default.nix`**

  ```nix
  # ZR-specific jira configuration module.
  #
  # Enables the generic `jira` CLI (from phillipgreenii-nix-base homeModules.jira)
  # and writes ~/.config/jira/config.toml with the ZipRecruiter tenant values.
  #
  # The API token is resolved at runtime via the macOS Keychain:
  #   security find-generic-password -s zr-jira -a <email> -w
  # The secret NEVER enters the nix store.
  #
  # One-time setup: see docs/superpowers/plans/2026-06-26-generic-jira-access-tool-sp2-zr-edge.md
  # Task 4 (keychain migration).
  { lib, pkgs, ... }:
  let
    email = "phillipg@ziprecruiter.com";
    baseUrl = "https://ziprecruiter.atlassian.net";

    configToml = pkgs.writeText "jira-config.toml" ''
      base_url = ${lib.strings.escapeNixString baseUrl}
      email    = ${lib.strings.escapeNixString email}

      default_limit = 100

      [secret]
      source  = "command"
      command = [
        "security",
        "find-generic-password",
        "-s", "zr-jira",
        "-a", ${lib.strings.escapeNixString email},
        "-w",
      ]
    '';
  in
  {
    # Enable the generic jira CLI (adds pkgs.jira to home.packages).
    phillipgreenii.jira.enable = true;

    # Write the ZR-specific config file. The store path is read-only; the jira
    # CLI opens it at startup to load non-secret config. source = "copy" so
    # home-manager makes a mutable user-owned copy at activation time (not a
    # symlink into the read-only store, which would prevent the CLI from
    # creating a sibling lock file if it ever needs one).
    xdg.configFile."jira/config.toml" = {
      source = configToml;
    };
  }
  ```

  > **Note on `lib.strings.escapeNixString`:** this produces a double-quoted string with
  > escaping for any special characters. The resulting TOML is valid because TOML also uses
  > double-quoted strings. The `command` array uses the TOML inline-array syntax; confirm the
  > generated file with `cat ~/.config/jira/config.toml` after the first apply.

### Task 3: Back-compat alias — `pg-pr-issues-jira-zr` forwarding wrapper

**Files:**

- Modify: `modules/pg-pr-zr/default.nix`

**Context:** `pr-pool` shells out to `pg-pr-issues-jira-zr search --jql … --limit N` and expects
the `{items, truncated}` JSON envelope. The current `pg-pr-zr` module installs a wrapper that:

1. Sets `JIRA_BASE_URL`, `JIRA_EMAIL`, and reads `JIRA_API_TOKEN` from a token file.
2. `exec`s the real `pg-pr-issues-jira-zr` binary from `cfg.jira.realBinary`.

After SP2, the real binary disappears (it is superseded by the generic `jira`), but the wrapper
name must stay. The new wrapper is simpler: just `exec jira "$@"`. Because `jira` is on PATH (via
`phillipgreenii.jira.enable = true`, installed in Task 2) and its config is at
`~/.config/jira/config.toml`, no env var injection is needed in the wrapper.

The existing `zr.pgPrZr.jira` option block in `machines/phillipg-mbp-02/default.nix` (lines
62-72) configures `baseUrl`, `email`, `tokenFile`, and `realBinary`. After SP2 these values are
superseded by the generated TOML. The options can be removed from the machine config once the
wrapper is replaced, or left in place if the module still declares them (with `mkIf
jiraWrapperEnabled` gating their use). See Open Design Decision 2.

**Minimal path (recommended):** replace the `jiraWrapper` derivation in
`modules/pg-pr-zr/default.nix` with a new derivation that simply delegates to `jira`, and
update the install condition to no longer depend on `tokenFile`.

  ```markdown
  # SP2 Keychain Setup — One-Time Migration

  After applying the SP2 home-manager config, run the following steps ONCE on the machine.

  ## Context

  The generic `jira` CLI resolves the Atlassian API token by running:
  ```

  security find-generic-password -s zr-jira -a phillipg@ziprecruiter.com -w

  ````

  The unified service name `zr-jira` replaces two divergent names used by the old tools:

  | Old tool                  | Old keychain service name          |
  | ------------------------- | ---------------------------------- |
  | `work-activity-tracker`   | `work-activity-tracker-jira`       |
  | `activity-collector`      | `activity-collector-jira`          |
  | `pg-pr-issues-jira-zr`    | token file at `~/.jira_api_token`  |

  ## Prerequisites

  - You have a valid Atlassian API token for `phillipg@ziprecruiter.com`.
    Generate one at <https://id.atlassian.com/manage-profile/security/api-tokens> if needed.

  ## Step 1: Add the unified keychain entry

  ```bash
  security add-generic-password \
    -s zr-jira \
    -a phillipg@ziprecruiter.com \
    -w
  ````

  `security` will prompt for the password — paste your API token. The `-w` flag reads from
  stdin interactively (do NOT pass the token as a command-line argument; it would appear in
  shell history).

  ## Step 2: Verify the entry

  ```bash
  security find-generic-password -s zr-jira -a phillipg@ziprecruiter.com -w
  ```

  Expected: prints your API token followed by a newline, exits 0.

  ## Step 3: Verify `jira auth-status`

  ```bash
  jira auth-status
  ```

  Expected output: `OK` (exit 0). If you see `UNAUTHENTICATED` (exit 5), the token is
  incorrect. If you see `MISSING` (exit 3), the keychain entry is absent or has the wrong
  service/account name.

  ```bash
  pg-pr-issues-jira-zr search --jql "project = FINDEV AND status = \"In Progress\"" --limit 3
  ```

  Expected: JSON `{"items":[...],"truncated":...}` with up to 3 items.

  needed:

## Open Design Decisions — RESOLVED (2026-06-29)

Resolved with the user; see bd memory `jira-tool-cross-cutting-decisions`. Summary: **#1 → TOML**
(decision #5); **#2 → Path A, keep `modules/pg-pr-zr` as the alias shim**, removal deferred to a
follow-up bead; **#3 → Option A runbook** (`docs/superpowers/plans/2026-06-29-jira-zr-keychain-setup.md`);
**#4 → `secret.source = "command"`** against the unified `zr-jira` keychain entry; **#5 → retain the
`pg-pr-zr-build` check** (harmless); **#6/#7 → no change needed.** Implementation used
`(pkgs.formats.toml {}).generate` (robust, no hand-rolled escaping) per #7's note that the repo
already uses `formats` generators. Detail retained below for traceability.

### 1. Config delivery: nix-generated TOML at `$XDG_CONFIG_HOME/jira/config.toml` vs. env vars via wrapper

**This plan chooses the TOML approach.** The `jira` CLI reads
`$XDG_CONFIG_HOME/jira/config.toml` by default (via `defaultConfigPath()` in `cmd/jira/main.go`),
so no wrapper is needed to inject env vars. The TOML is generated by nix as a store derivation
and linked via `xdg.configFile`, keeping all non-secret config in one place (no env var scattering
across multiple shell init files).

**The env-var alternative** (export `JIRA_BASE_URL`, `JIRA_EMAIL`, `JIRA_API_TOKEN_CMD` from a
shell init snippet) would avoid writing any file to disk, but:

- The generic CLI does not support a `JIRA_API_TOKEN_CMD` env var (only `JIRA_API_TOKEN` for the
  literal token value); replicating the `command` source via env vars would require a wrapper
  script anyway.
- The TOML approach is already used by `pn` (repo-base's other Go CLI) and is idiomatic for the
  workspace.

**Decision: TOML.** If the `XDG_CONFIG_HOME` or `HOME`-derived path ever conflicts with another
tool, use `--config ~/.config/jira/config.toml` via an alias or shell init snippet.

### 2. Whether to retire `modules/pg-pr-zr` entirely or keep it as the alias shim

**Status: open.** Two paths:

**Path A (this plan's choice): keep `modules/pg-pr-zr/default.nix` as the alias shim.** Replace
`jiraWrapper` with `jiraAlias` (Task 3). The module stays in the import list (`machines/phillipg-mbp-02/default.nix:49`). The `zr.pgPrZr.jira` options are deprecated but remain
to avoid a machine-config breakage. The Pattern-B `pg-pr-zr` build (`build.nix`) is kept because
the `pg-pr-zr-build` flake check asserts it and removing it is a separate cleanup task.

**Path B: retire the module after confirming the alias is live.** Once SP2 is applied and the
alias is verified working, remove `modules/pg-pr-zr/default.nix` and `build.nix`, remove the
import from the machine config, and remove the `pg-pr-zr-build` flake check. This is cleaner but
is a non-trivial change (the Pattern-B build is load-bearing for `pg-pr-cicd-captains-log` as
well as `pg-pr-issues-jira-zr`). Unless `pg-pr-cicd-captains-log` is also superseded by SP2, the
module cannot be retired yet.

**Recommendation:** proceed with Path A. Track Path B cleanup (including `pg-pr-cicd-captains-log`
disposition) as a follow-up bead.

### 3. Keychain migration step ergonomics (the one-time `security add-generic-password` step)

The runbook in Task 4 documents the manual `security add-generic-password -s zr-jira -a
phillipg@ziprecruiter.com -w` command. Three ergonomic options:

**Option A (this plan):** a runbook document the user follows once. No automation; explicit.

**Option B:** a nix check that warns (but does not fail the build) if the keychain entry is
absent. Implemented as a `pre-activation` script via `system.activationScripts`. This is
complex and `security` is interactive — it cannot run unattended.

**Option C:** a helper script (`Scripts/jira-setup.sh`) that wraps `security
add-generic-password` with a prompt and verification step. Lower friction than a raw runbook
command.

**Decision: Option A** for SP2. If keychain setup friction becomes an issue during SP5/SP6
migrations (other tools adopting the `zr-jira` service name), revisit Option C.

### 4. Whether `secret.source = "command"` (keychain) or `"file"` (reuse existing tokenFile) is preferred for ZR

**This plan chooses `secret.source = "command"`.** Rationale:

- The token file at `~/.jira_api_token` is a holdover from the old `pg-pr-issues-jira-zr`
  wrapper. It contains a plaintext secret; if it were to be included in a nix derivation path
  it would enter the store. The file path is kept out of the store by declaring it as a
  `types.str` (not `types.path`) in the module option, but this adds cognitive overhead.
- The macOS Keychain is the correct secret store on this platform. The `command` source is the
  spec's stated mechanism for keychain access (§8.3 and §8.4). The two existing activity-tracker
  tools already use this pattern.
- The `file` source (`secret.source = "file"`) is useful when a secret manager (e.g., Vault,
  1Password CLI) writes the token to a file at a well-known path. That pattern is not established
  in this workspace for Jira; the Keychain is.

If the user wants to keep using the token file as a transitional step (before running the
`security add-generic-password` command), they can manually override `secret.source` to `"file"`
and `secret.path` to `"~/.jira_api_token"` via a `--config` override or env var, without
changing the nix-managed TOML.

### 5. `pg-pr-zr-build` flake check retention

The `pg-pr-zr-build` check in `flake.nix:531-542` builds the Pattern-B derivation and asserts
both `pg-pr-cicd-captains-log` and `pg-pr-issues-jira-zr` binaries are present. After SP2, the
`pg-pr-issues-jira-zr` binary in `pgPrZr` is no longer the active one (the alias wrapper is),
but the check still passes because the old binary is still built. This is harmless.

**Decision: retain the check** in SP2 to avoid disturbing the build contract. If/when
`modules/pg-pr-zr/` is retired (Open Decision 2, Path B), the check must be removed at the same
time.

### 6. `install-metadata` update for the nix-base input bump

`darwin/system/default.nix:60-86` lists the flakes for which install-metadata files are written.
`phillipg-nix-repo-base` (`phillipgreenii-nix-base`) is already in that list (line 65). The
input bump in Task 0 changes `phillipgreenii-nix-base`'s rev, which changes the version string
in `~/.nix-profile/share/pn/phillipg-nix-repo-base-install-metadata.json`. This is the intended
behavior — no code change needed.

### 7. `xdg.configFile` source vs. text

The plan uses `source = configToml` (a `pkgs.writeText` derivation). An alternative is
`text = builtins.readFile configToml` or writing the TOML directly as a `text = ''...''`
string. Both work; `source = derivation` is the more common pattern in this repo's home modules
(confirmed: `machines/phillipg-mbp-02/default.nix:236` uses `source =
(pkgs.formats.yaml {}).generate …`). Use `source` for consistency.
